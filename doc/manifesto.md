# Manifesto

AlifePlus is aligned with GSC's original A-Life vision and the X-Ray engine architecture it was built on. The original A-Life was an event-driven simulation where NPCs lived autonomous lives -- found items, traded, responded to threats, and pursued goals independently of the player. Most of it was cut before Shadow of Chernobyl shipped. What remained was smart terrains and campfires.

Every design choice traces back to the original engine architecture or to documented GSC developer intent. Nothing is invented and nothing is faked.

---

## Principles

### Event-Driven Contract

Nothing runs unless the engine says something happened. Callbacks fire, predicates evaluate, consequences dispatch. If nothing happens in the simulation, the mod consumes zero cycles.

- `game_object_space.h` -54 callback types defined
- `entity_alive.cpp` -`eDeath` Lua callback on entity death
- `xrServer_process_event.cpp` -`GE_DIE` net event propagation
- `alife_simulator_base2.cpp` -`alife().on_death` processes A-Life death
- `alife_online_offline_group.cpp` -`notify_on_member_death` updates the squad
- `alife_monster_detail_path_manager.cpp` -`on_location_change` fires on smart terrain transition
- `alife_monster_abstract.cpp` -commented-out `alife().check_for_interaction(this)`, marked `#pragma todo("Do not forget to uncomment here!!!")`. NPC-NPC encounter hook during graph traversal, never connected

> "A character's life now worked in such a way, that he took the quest generated by some smart terrain, then walked to the destination it had set for him. After getting there, he would get points, and the smart terrain would take him under control for some time."
> -Dmitriy Iassenev, Game Developer (2008) [1]

### Physical Simulation Guarantee

Nothing is spawned, teleported, or created to fill a role. Every entity involved in any consequence was already there, living its own A-Life. This is an invariant, not a guideline.

- `alife_monster_brain.cpp` -brain selects smart terrain target
- `alife_monster_detail_path_manager.cpp` -A* graph traversal, distance from `speed * time_delta * time_factor`
- `alife_graph_registry_inline.h` -atomically updates entity graph ID, position, and node on level change
- `alife_online_offline_group.cpp` -squad syncs all member positions to commander
- `alife_switch_manager.cpp` -online/offline switching, distance-based with hysteresis

> "The player is not the most powerful or the strongest -he is one of many, whose lives crossed the Zone."
> -Alexey Sytianov, Shpil Magazine [2]

### Symmetric Entity Treatment

Causes fire for all creatures equally and consequences resolve for all factions equally. The simulation does not revolve around the player.

Traders, quest givers, companions, mechanics, barmen, leaders, guides, and story characters are filtered out before any squad selection -over 80 named characters protected through a shared predicate in xlibs. Quest targets are not protected. Your target gets in a fight and dies before you reach him, and you fail the quest. Protecting quest targets from the simulation would mean faking it.

You are not special.

## Architecture

### Event Pipeline

Classical pub/sub chain: engine callbacks and synthetic hooks feed into a producer that gates events through a budget distributor, on-map resolver, Bresenham ratio gate, class-isolated cooldowns, per-cause rate limit, and a predicate. Causes and consequences are fully decoupled -causes publish, consequences subscribe, neither references the other.

### Radiant Evaluation

Squads evaluate their surroundings when they transition between smart terrains. A squad arrives at a new location and looks around for stashes, unguarded outposts, threats, or unmet needs. All radiant causes originate here -stash, area, and needs.

- `on_location_change` -fires on every graph node transition
- `squad_on_enter_smart` -Anomaly hook, squad arriving at smart
- `squad_on_leave_smart` -Anomaly hook, squad leaving smart
- `can_choose_alife_tasks` -engine guard on task selection
- `smart_terrain_choose_interval` -time gate on smart selection from INI

> "I was very happy with the work done when I followed a stalker from one level to another, saw how he searched for artifacts, found them, returned to the dealer, approached him, traded, picked a new quest, went on -too bad this did not make it into the original game."
> -Dmitriy Iassenev, Game Developer (2008) [1]

### Synthetic Callbacks

Where the engine lacks a callback, a runtime hook emits a synthetic event without modifying base scripts.

- `AddScriptCallback` -declares a new callback name
- `SendScriptCallback` -dispatches to all registered handlers
- `_G` module table replacement -hook wraps existing function, calls original, emits new event
- Anomaly uses the same mechanism for `CPseudoGigant` stomp/hit callbacks, net spawn/destroy lifecycle hooks, and the bullet callback chain

### A-Life Ratio

30 maps simulate simultaneously. The same causes fire and the same consequences resolve on every map. The rules are identical. The frequency is not.

- `alife_movement_manager_holder.h` -`m_fGoingSpeed` for background levels, `m_fCurrentLevelGoingSpeed` for current level. Anomaly sets both to 3 in `m_stalker.ltx`, removing the differential
- `alife_update_manager.cpp` -`update_switch` runs current-level entities every render frame with real-time budget. `update_scheduled` round-robins all offline entities at fixed count per tick
- `alife_graph_registry.cpp` -`setup_current_level` creates dedicated level registry with own processing budget
- Background events outnumber current-level events ~50:1

A-Life Ratio is a Bresenham ratio gate restoring this differential at the cause evaluation layer. Default yields 4:1 admission ratio. Slider from -10 (background only) to +10 (current level only).

## Mechanics

Every mechanic below: observe an engine event, evaluate a predicate, dispatch a consequence using real entities already in the simulation.

### Massacre: Scavenge, Investigate

When deaths accumulate at a location, nearby scavenger factions converge on the site and the victims' faction sends squads to investigate.

- `notify_on_member_death` -fires on squad member death
- `monster_corpse_memory` -scans for dead entities, tracks by distance, returns nearest through `find_best_corpse`
- `state_defs.h` -full state chain: approach, inspect, drag, eat, rest. Corpse locking prevents competition. Individual monsters already converge on corpses -scavenge scales this from one body to a massacre site
- `relation_registry` -tracks kill-based attitude shifts between factions, nothing acts on the change -investigate acts on it

> "To notify player about the offline activity, we used news, which were generated if some character could see an event or its consequences."
> -Dmitriy Iassenev, Game Developer (2008) [1]

### Squad Kill: Revenge, Flee

When the last member of a squad dies, same-faction squads pursue the killer. Nearby squads of the victim's faction fall back to the nearest base.

- `relation_registry_actions` -adjusts faction goodwill on kills, scaled by victim rank. Engine tracks who killed whom and shifts attitudes, nothing acts on the change -revenge acts on it
- `ai_stalker` -danger evaluation states trigger retreat behavior
- `sim_board` -routes squads to faction bases through precondition gates -flee uses same routing to nearest friendly base

> "One of the goals that developers set for themselves when creating the mechanics of the Faction War is for the player to see what his actions lead to."
> -Clear Sky pre-release interview [5]

### Base Kill: Support, Flee

When a faction base sustains enough deaths, nearby friendly squads converge to reinforce and squads at the base evacuate.

- `sim_board` -gates squad-to-base assignment through faction preconditions
- `general_base_precondition` -faction and time-of-day checks for base access
- Clear Sky faction war routed reinforcements to contested control points -support uses the same precondition routing when base death threshold is breached

> "The in-game Zone has been shaken by furious war of factions for the new territories, resources, and control points of the important paths to the center of the Zone."
> -Clear Sky official gameplay page [4]

### Elite: Promote

An NPC accumulates kills and levels up through 10 tiers, gaining AP ammo, grenades, and engine rank boosts.

- `character_info_defs.h` -`CHARACTER_RANK_VALUE`, Russian comment: "from 0 (completely inexperienced) to over 100 (very experienced)"
- `relation_registry_actions.cpp` -`rank_kill_points`, 5 for novice kill up to 20 for legend kill
- `ai_stalker.cpp` -Russian comment: "compute immunities depending on rank". Rank 0-100 interpolates dispersion, immunity, visibility. Anomaly sets immunity and visibility to 1.0/1.0 (dead code), dispersion 1.0/0.8 (20% accuracy at max rank)
- `k_ap` -armor penetration coefficient per weapon. Experienced NPC should use best available `k_ap` for weapon class
- Combat AI has grenade usage built in, but NPCs exhaust initial supply permanently. No NPC throws a grenade past early game without restocking

### Elite Kill: Bounty, Targeted

When an elite dies, the killer receives rubles scaled by the elite's level and other elites on the same tier pursue the killer.

- `rank_kill_points` -engine rewards kills with rank points. Bounty extends this with rubles
- `CHARACTER_RANK_VALUE` -engine tracks kill-based progression per NPC but has no consequence for killing high-rank entities. Targeted adds that consequence

### Wounded: Hunt, Help

When an NPC or the player uses a healing item, nearby predator mutants close in and nearby same-faction squads rush to help.

- `state_defs.h` -`eStatePredator` with ambush states, `eStateAttack` for engagement, `eStateEat` with full corpse lifecycle (approach, inspect, drag, eat, rest)
- `chimera_state_hunting` -dedicated pursuit and ambush behavior
- `monster_corpse_memory` -drives convergence on dead entities. Predator state machine targets dead entities -wounded extends target from dead to weakened prey
- `base_monster.h` -`GetSatiety()` returns hardcoded 0.5, `ChangeSatiety()` does nothing. Hunger-driven hunt cycle planned and abandoned, predator states survived
- `xr_eat_medkit.script` -heals NPC directly, fires no event. No `npc_on_item_use` callback exists in Anomaly. Synthetic callback hooks handler, emits `x_npc_medkit_use`

> "A complex AI system with a life simulation system that enables any monster and NPC to act independently on any game level, with creatures having a full life cycle where they hunt, feed, take rest, and sleep."
> -Oles Shishkovtsov, ModDB interview [3]

### Harvest: Hunt

When an NPC or the player picks up an artefact, nearby outlaws pursue the carrier through real movement.

- `InventoryOwner.cpp` -`eOnItemTake` fires for both actor and NPCs
- `xr_gather_items.script` -NPC artefact gathering with community filters and timers
- `xrServer_Objects_ALife_Monsters_script.cpp` -`spawn_artefacts` commented out in engine source. Artefact economy planned at engine level, competitive response never added

> "In early versions of the simulator, a character knew one or several dealers, which had sets of quests that they generated based on the map of anomalous activity and requests from organizations, which wanted to get rich from artifacts found in the Zone."
> -Dmitriy Iassenev, Game Developer (2008) [1]

### Stash: Loot, Ambush, Fill

A squad discovers a nearby stash on smart terrain transition, walks there to loot it, camp to ambush others, or hide supplies inside.

- `game_object_space.h` -`eInvBoxItemTake`, actor-only callback for inventory box interaction
- `treasure_manager.script` -indexes all inventory boxes on load, fills half with random loot, tracks for player. NPCs never interact with stashes
- `axr_main.script` -`npc_on_item_take_from_box` callback exists, handler is empty
- NPC stash involvement is purely narrative: `dialogs.script` (surrender reveals locations), `xr_effects.script` (task rewards mark on map), `ui_pda_npc_tab.script` (PDA hints at contents). Infrastructure built and left disconnected

> "The character would take on a quest to search for the artifact, go to the approximate area in which it could be found, if able to find the artifact, brought it back to the dealer, traded it, then picked a new quest."
> -Dmitriy Iassenev, Game Developer (2008) [1]

### Area: Conquer

When a squad discovers an unguarded smart terrain nearby, it walks there and claims it on arrival.

- `xrServer_Objects_ALife.h` -`CSE_ALifeSmartZone`, bare scheduling shell with zero faction fields. All territory control is Lua-side
- `smart_terrain.script` -`check_smart_faction` derives faction from physically present stalkers
- `sim_board.script` -`general_base_precondition` (faction + time gate 3:00-6:00), `general_territory_precondition` (faction + time gate 9:00-15:00). Squads never proactively decide to claim territory
- On arrival: handler sets `faction_controlled` and `respawn_params` at runtime so engine `try_respawn` spawns conqueror's faction. Conquered smarts tracked with FIFO eviction, persist across save/load

> "A control point leading to a place with artifacts is controlled by bandits. Their destruction means that the path to resources is safe."
> -Clear Sky pre-release interview [5]

> "The more points with artifacts the faction controls, the cooler stalkers it can recruit."
> -Clear Sky pre-release interview [5]

### Needs: 9 Drives, 15 Consequences

When a squad transitions at a smart terrain, it evaluates nine human needs and pursues whichever is strongest.

Drive scoring uses Hull's Drive Reduction Theory (1943) weighted by Maslow's hierarchy of needs (1943). `drive = weight * (elapsed / threshold)^2`. Weights follow Maslow -survival outweighs security outweighs belonging. The squared deprivation ratio adapts Hull: drive grows nonlinearly with deprivation, so mild discomfort is noise and critical need dominates.

Hull and Maslow were the dominant behavioral models when GSC designed A-Life in 2002-2004.

The nine drives:

- **Hunger** -finds a campfire, eats from inventory (bread, sausage, canned goods)
- **Sleep** -waits for nightfall, finds a campfire, sleeps through the night
- **Rest** -finds a campfire, smokes a cigarette, has a drink
- **Heal** -returns to a friendly base, uses a medkit, bandage, or stimpack
- **Shelter** -returns to a friendly base when exposed too long
- **Money** -searches anomaly fields for artefacts, or hunts mutant lairs
- **Supply** -visits a trader, exchanges an artefact for ammo, grenades, or medical supplies
- **Job** -guards their base, explores the Zone, researches anomalies. Monolith and Greh worship. Military and Duty drill
- **Social** -finds a campfire or returns to base, shares cigarettes and drinks

NPCs consume inventory items on arrival -food, cigarettes, drinks, medkits, or artefacts depending on the need.

- `base_monster.h` -`GetSatiety()` returns hardcoded 0.5, `ChangeSatiety()` does nothing. Hunger planned and abandoned. Continuous deprivation variable = Hull drive strength
- `state_defs.h` -`eStateEat` (approach, inspect, drag, eat, rest), `eStateRest` with Sleep. States exist, nothing triggers them by need
- `sr_camp.script` -campfire social dynamics at smart terrains
- `gulag_general.script` -assigns campfire, rest, guard jobs
- `xr_eat_medkit.script` -NPC healing handler
- Job system, social behavior, and health response all exist. No algorithm connects needs to behavior

> "It was planned that character behaviour would diversify due to introduction of requirements of food and sleep."
> -Dmitriy Iassenev, Game Developer (2008) [1]

> "Now, instead of creation of an algorithm for choice of current needs and their satisfaction, a simpler model has been used, which offloaded this work to the smart terrains."
> -Dmitriy Iassenev, Game Developer (2008) [1]

> "I would very much like to play a game in which characters would live their own lives, each would have his own goal in the game, each would have human-like (or some specific for monsters) needs that the character would have to satisfy."
> -Dmitriy Iassenev, Game Developer (2008) [1]

**Item consumption.** `GetSatiety()` returning 0.5 proves hunger was planned. `eStateEat` proves eating was planned. Walking to a campfire hungry and eating nothing is fake.

**Supply exchange.** The artefact economy in [1] required exchange. Free items from a trader is a spawn, not a trade.

### Personality

GSC modeled personality with independent attribute axes that fed probability lookup tables for behavior decisions. Five axes from the design documents [8], a moral axis with eight character types, and faction-level personality as a core concept. None of it shipped.

GSC's five personality axes [8]:

- **PersonalAggressiveness** (1-5) -- willingness to engage. `Expediency` function input: aggression=1 yields 5% action probability, aggression=5 yields 65%
- **PersonalGreed** (1-5) -- profit motivation. Combined with aggression in `Expediency`: high greed + low aggression = rob when safe; both high = attack for loot
- **PersonalIntelligence** (1-5) -- retreat decision quality. `EnemyRetreatProbability = f(detectability, intelligence)` -> 20-65%
- **PersonalEyeRange** (1-5) -- awareness. `EnemyDetectProbability = f(detectability, eye_range)` -> 1-90%
- **PersonalRelation** (1-2) -- friend/enemy. Caps `Expediency` at 45% for friends, 65% for enemies

GSC's moral axis classified characters into eight types (principled/neutral/unprincipled x good/cold/evil) [8]. Each type defines specific behaviors: looting, betrayal, revenge pursuit, flee distance, mutant attitude, SOS response.

> "Each faction possesses a distinct 'personality' (either pre-assigned or randomly generated); based on this personality, the faction determines which other factions it will initially treat as allies or enemies." [8]

> "Based on distance, potential gain, and the character's personality, a course of action is selected: (a) Ignore the signal. (b) Kill and loot the sender. (c) Rescue the sender." [8]

> "A character's personality determines what they prioritize as more valuable: money or reputation points (including negative reputation points)." [8]

The `Expediency` function is the strongest evidence: a 4D lookup table `f(Relation, Cost, Greed, Aggression)` -> action probability (5-65%). The decision is a function of who you are, not a coin flip. GSC also treated mutants as actors in the same system -- the eight personality types define mutant-related behavior explicitly: "Tries to avoid killing mutant animals whenever possible" (Principled Good), "Ignores mutant animals unless they display aggression" (Cold-blooded), "Deliberately eliminates humanoid mutants" (Principled Good) [8].

AlifePlus implements this at faction level. Seven personality fields with per-faction multipliers replace random chance gates. Both stalker and mutant factions share the same table, same lookup, same gate. The decision is deterministic given the faction and the event.

| Field | GSC design docs | X-Ray engine | AP consequences |
|-------|----------------|--------------|-----------------|
| aggression | PersonalAggressiveness, `Expediency` [8] | `m_bAggressive`, `base_monster_misc.cpp` | revenge, elite hunt, wounded hunt, harvest hunt |
| greed | PersonalGreed, `Expediency` [8] | -- | stash loot, stash ambush, stash fill, massacre scavenge |
| survival | `EnemyRetreatProbability` [8] | `panic_threshold`, `CMonsterMorale`, `monster_morale.cpp` | flee from squad kill, flee from base kill |
| perception | PersonalEyeRange, `EnemyDetectProbability` [8] | -- | massacre investigate |
| territory | Faction territorial expansion, `Faction_settings.htm` [8] | `CMonsterHome`, `monster_home.cpp` | area conquer |
| relation | PersonalRelation, `Expediency` [8] | -- | base kill support, wounded help |
| discipline | Principled/unprincipled moral axis [8] | -- | all 16 needs consequences |

**aggression** -- engine mirrors GSC's axis with `m_bAggressive`, set from enemy count, sounds, and hits. For mutants, maps to inverted `panic_threshold`: chimera=0.0 (cannot panic), boar=0.5, rat=0.9 (flees easily).

**greed** -- profit-driven action selection via GSC's `Expediency`. Gates resource-acquisition: looting, ambushing, stash filling, massacre scavenging.

**survival** -- GSC's retreat probability + engine morale system (`monster_morale.cpp`): morale drops on hits, triggers `eStatePanic` below `panic_threshold`. High survival = faction retreats when losing.

**perception** -- GSC's eye range fed detection probability (1-90%). Gates investigation and awareness consequences.

**territory** -- engine `CMonsterHome`: three concentric radii, `m_aggressive` flag for home defense. Gates area conquest.

**relation** -- GSC's binary friend/enemy input capped `Expediency` for friends. Gates solidarity: base reinforcement, wounded aid.

**discipline** -- from GSC's principled/unprincipled axis. Gates all 16 needs consequences: routine behavior (eating, sleeping, guarding, socializing).
